# 旧版浏览器兼容模式设计

## 目标

为旧版本浏览器提供一个可自动启用的兼容模式，确保页面在 IE、旧版 Edge、旧版 Firefox、旧版 Chrome 中至少可以稳定展示和使用左侧基础导航能力。

兼容模式的优先级是“可打开、可识别、可点击、可跳转”，而不是保留当前视觉效果或动态体验。

## 成功标准

- 旧版浏览器可自动进入兼容模式，不要求用户手动切换。
- 兼容模式下保留头像、姓名、状态、社交链接、页脚。
- 兼容模式下移除 slogan 区域和整个右侧面板。
- 兼容模式下大部分动画和特效被关闭，页面不依赖现代浏览器 API 才能正常展示。
- 页面在兼容模式下仍能正常访问社交链接，不因图标、壁纸、观察器、滤镜等失败而不可用。

## 非目标

- 不追求在旧版浏览器中完整还原当前双栏视觉设计。
- 不为壁纸、滚动动画、打字机、磨砂玻璃等高级效果补齐 polyfill。
- 不新增单独的旧版页面入口，例如 `compat.html`。
- 不承诺像素级一致性，只保证基础内容和导航可用。

## 现状摘要

当前站点是静态双栏个人主页：

- `index.html` 直接加载 `config.js` 和 `js/*.js` 全局脚本。
- 左侧包含头像、姓名、状态、slogan、社交链接、页脚。
- 右侧包含时间组件、壁纸滚动区、移动端面板切换。
- `js/main.js` 统一初始化 Font Awesome、壁纸模块、滚动动画、移动端交互、基础模块。
- `css/style.css` 使用了大量现代 CSS 能力和动画。

对旧浏览器最脆弱的点包括：

- `class`、`async/await`、箭头函数等现代 JS 语法。
- `IntersectionObserver`。
- CSS 自定义属性、`clamp()`、CSS Grid、`backdrop-filter`。
- 动态图标加载和大量基于动画/过渡的交互。

## 总体方案

采用“自动检测 + 单页兼容样式/逻辑降级”的方案。

实现方式：

1. 在页面尽早执行一个轻量兼容检测脚本。
2. 若命中旧浏览器条件，则给根元素加上兼容模式标记，例如 `legacy-compat`。
3. CSS 使用该标记覆盖布局与视觉表现，只保留左侧基础信息。
4. JS 初始化流程检测到兼容模式后，跳过不必要模块，仅运行基础配置渲染。

该方案避免维护第二套 HTML，同时可以让现代浏览器继续保留当前体验。

## 兼容模式下保留的功能

- 头像显示。
- 姓名显示。
- 状态文本显示。
- 社交链接渲染与跳转。
- 页脚文字显示。
- 简化后的加载提示，或直接显示基础内容。

这些是兼容模式的“最低可用面”。

## 兼容模式下移除或禁用的功能

- slogan / 打字机区域。
- 右侧时间与壁纸区域。
- 噪点叠加层。
- 动态加载 Font Awesome。
- 滚动触发动画。
- 悬停位移、阴影强化、粘性头像缩放等交互特效。
- 移动端壁纸切换按钮和关闭按钮。
- 依赖现代 API 的壁纸懒加载与自动滚动逻辑。

## 浏览器检测策略

兼容模式应使用保守的自动检测，只要命中明显高风险条件就进入降级模式。

建议检测条件：

- 检测 IE 特征：`document.documentMode`。
- 检测旧版 EdgeHTML 特征：`window.StyleMedia` 且不是 Chromium Edge 特征。
- 检测缺失关键能力：
    - 不支持 `IntersectionObserver`
    - 不支持 CSS 自定义属性检测
    - 不支持 `classList` 或其他关键 DOM 能力时直接进入兼容模式

原则：宁可更早降级，也不要让旧浏览器落入现代模式后出现白屏或不可交互状态。

## HTML 设计

`index.html` 保持单页结构，不拆页面。

新增内容：

- 在样式和主体脚本之前插入一个很小的内联检测脚本。
- 检测到旧浏览器时，给 `<html>` 增加兼容模式类名。
- 可选增加兼容提示类名，例如 `legacy-compat`，供 CSS 和 JS 共用。

不新增复杂的条件注释或独立模板。

## CSS 设计

在 `css/style.css` 中新增一段兼容模式覆盖规则，集中管理。

关键覆盖点：

- `.legacy-compat .right-panel { display: none; }`
- `.legacy-compat .bio-container { display: none; }`
- `.legacy-compat .noise-overlay { display: none; }`
- `.legacy-compat .wallpaper-toggle, .legacy-compat .close-panel { display: none; }`

兼容模式布局应退回简单单栏：

- `html` 和 `body` 使用自然高度，避免强依赖 `100vh` 带来的旧浏览器问题。
- `.container` 取消模糊、过滤、禁用状态，改为普通块级布局。
- `.left-panel` 取消固定宽度、复杂阴影和高级过渡，改为居中单列或普通流布局。
- 社交链接从复杂网格退回为简单的块状列表或基础多列布局。
- 动画、transition、transform 在兼容模式下尽可能全部关闭。

样式原则：

- 兼容模式优先保证清晰稳定，不追求当前风格的完整延续。
- 允许保留最基础的边框与配色，但避免依赖现代属性。

## JavaScript 设计

`js/main.js` 需要成为兼容模式的总开关。

建议流程：

1. 读取根元素兼容类名，生成 `isLegacyCompat` 判断。
2. 若为兼容模式：
    - 跳过 Font Awesome 动态加载。
    - 跳过壁纸模块初始化。
    - 跳过滚动动画初始化。
    - 跳过移动端壁纸面板切换和粘性头像交互。
    - 直接显示主内容并隐藏复杂加载层。
3. 无论是否兼容模式，仍保留：
    - `initSocialLinks()`
    - `applyProfileConfig()`
4. `initTypewriter()`、`initTime()` 在兼容模式下不执行。

## 社交链接降级策略

社交链接在兼容模式中仍然保留，因为这是左侧导航的核心功能。

降级策略：

- 若图标字体不可用，链接文本本身必须足够可读。
- 不依赖图标存在才能辨识链接用途。
- 不依赖 CSS Grid 才能排版；必要时退回成垂直列表。
- 外链行为保持现状，邮箱链接仍走 `mailto:`。

这要求 `social.js` 生成的结构在“无图标、无高级布局”情况下仍可使用。

## 加载态设计

兼容模式只保留简化加载体验。

推荐做法：

- 不再展示旋转动画、进度条、百分比等复杂内容。
- 若需要保留加载层，仅展示简单文本，例如“Loading...”。
- 由于兼容模式不再初始化壁纸预加载，加载层应尽快移除，避免无意义等待。

目标是避免旧浏览器卡在复杂加载态中。

## 配置层设计

`config.js` 仍然保持为主要内容配置入口，不新增复杂兼容专用配置对象，除非实现阶段发现确有必要。

本次兼容模式以代码层的自动降级为主，不要求用户配置。

如果实现中需要少量可配置项，优先考虑增加类似以下的简单开关：

- `compatibility.forceLegacyMode`
- `compatibility.disableEffects`

但默认不引入，先保持最小改动。

## 错误处理

- 如果旧浏览器无法支持壁纸、图标、观察器、滤镜等能力，不应报错阻塞整个页面。
- 兼容模式判断必须足够早，避免先执行不兼容逻辑后再失败。
- 若检测逻辑本身失败，应默认进入更安全的简化路径，而不是继续加载复杂特效。

## 测试与验证策略

仓库没有自动化测试，因此以静态校验和手动验证为主。

验证步骤：

- 运行 `npm run lint`
- 运行 `npm run format:check`
- 运行 `npm run build`
- 在现代浏览器中确认默认模式未被破坏。
- 使用开发者工具模拟兼容分支或临时强制兼容类名，确认兼容模式仅展示左侧基础内容。

重点验证点：

- 现代浏览器仍显示完整双栏页面。
- 兼容模式下右侧完全隐藏。
- 兼容模式下 slogan 不显示。
- 兼容模式下社交链接仍可点击。
- 兼容模式下不再初始化壁纸和时间模块。
- 加载层不会长时间停留。

## 实施边界

本次工作聚焦在“旧浏览器最低可用兼容模式”，不包含：

- 代码全面 ES5 化重写。
- 构建链引入 Babel 或额外转译流程。
- 全量支持所有历史浏览器的细节兼容。

如果实现后发现某些核心脚本仍因语法级不兼容而无法执行，则后续再评估是否需要对关键脚本做局部语法降级。

## 推荐实施顺序

1. 增加最早期兼容检测标记。
2. 在 CSS 中实现兼容模式的单栏静态布局。
3. 在 `main.js` 中按兼容模式跳过复杂模块初始化。
4. 校正社交链接在无图标/无复杂样式下的可用性。
5. 执行 lint、format、build 验证。

## 风险与取舍

风险：

- 当前部分脚本使用现代语法，若旧浏览器在解析阶段直接报错，仅靠运行时检测不一定足够。
- `style.css` 中大量现代 CSS 可能在旧浏览器下被部分忽略，需确保兼容覆盖足够直接、足够靠后。
- 图标缺失后视觉会变朴素，但这是可接受取舍。

取舍：

- 选择最小兼容面，放弃右侧视觉内容和多数动效。
- 选择自动降级而不是引入复杂 polyfill 或双页面维护。
- 以稳定可用为目标，而非完整保真。
